The OSM Programming Language

By: Anna Clayton, Michael Cohoon, and Sean Dunn

Motivation

  • OSM stands for "Old Spice Man" and is pronounced "Awesome"
  • When?
    • Created Spring 2012 for MSCS 610: Advanced Theory of Programming Languages (duh).
  • Why?
    • Programmers tend to lack a sense of humor when they read or write code. We wanted to force them into a chuckle.
  • Who?
    • Programmers with a sense of humor.
  • What?
    • A general purpose language suitable for writing command line scripts, system administrator tasks, and possibly templating.

Influences

  • Scala, Java, and other JVM languages

Syntactic features - Data Structures

  • Objects! – OSM is a heavily class-based, object-oriented Language
  • We support Number, Boolean, etc. through built-in objects
    • Number Literal
      • -5, 10.13, 1337.1337f
    • String Literal
      • "Heeellllooooo!", "Why hello there #(:person.name)#"
    • Boolean Literal
      • true, false
    • Array Literal
      • [1,2,3,4]
      • ["This", "is", "an", "Array"]
    • Dict Literal
      • {'key':'value'}
      • {'name':'Tinman', 'address':'1 Woodcutter Lane', 'country':'Oz'}

Syntactic features - Variables and Such

  • Creating a variable and setting a variable are two explicit statements (this is mainly for the lolz)
    • Creation: its :x the Number
    • Setting: the :x is now 42
  • We also have Type checking, casting, and all that good stuff
    • Type Checking: :a_man isa Person
    • Casting: did you know that :a_lady is LadyScentedBodyWashUser
    • Negation: not :some_boolean_value

Other Cool Things

  • Uses Scala's idea that everything's an object and operations are really just functions
    • 5 + 10 is the same as 5.+(10)
    • This allows for the creation of really cool DSLs
    • For example, :the_gazelle eats :grass is far eaiser to understand than :x.eats(:y)
  • DIAMONDS, TWOTICKETS
  • Awesome base classes: LadyScentedBodyWashUser, ManYourManCanSmellLike
  • %%Comments%%
  • Multi Setters: the :x, :y, :z are now true, "Silverfish Hand Grab", 123.0
  • When your code smells like OSM code you don't need to handle exceptions
    • But you could add this in to the language at a future date.

Simplicity

  • OSM
    • By definition, no. OSM is not writably or readably simple.
    • This is due to the presence of...I don't even know what to call it...SUPER OPERATORS
  • Scala
    • Simpler than OSM, but not as simple as Python
    • Concurrency model complicated the overall simplicity of the language
    • Common function names "tend" to perform common operations
  • Python
    • Very simple
    • Becoming very popular as a teaching language due to its easy to understand syntax and operations

Orthogonality

  • OSM
    • Yes!
    • This comes from the combination of the Module and Class systems which allows developers to write modular, reuasble solutions
  • Scala
    • Yes!
    • Scala's Orthogonality differs slightly from OSM's in that it is more Orthogonal, due to the concept of Objects (singletons), companion classes, and - in a way - Actors
  • Python
    • Yes!
    • Python's Orthoganality comes from its roots in the functional mindset and the modular code concept that is common of "The Python Way"
    • For example, the design idea that a module with a set of abstract, common functions is better than a class with the same over a resctricted set of data members

Control Statements

  • OSM
    • We previously touched upon how OSM only has one explict control structure that can function in the same fashion as some other types of structures
    • We stole this idea from Google's Go Language and their for() loop
  • Scala
    • Scala has both the familiar for() and while() loops that function in a similar fahsion to Java's
    • It also has the Python idea of List Comprehensions and common functional operations such as map, fold, and filter
      • for (x <- List.range(0,10)) yield x*x
  • Python
    • While python has both a for and while loop, the main Python loop that is most common - for - is actual more like a foreach loop
      • for item in items: ...
    • Python also has list comprehensions - or loops that return a list of items
      • squares = [x**2 for x in range(10)]

Data Types and Structures

  • OSM
    • We showed these earlier, but a quick recap is that everything is an object
    • We have the following literals: Number, String, Boolean, Array, Dict
  • Scala
    • Scala has all the same types as the Java primitives, as well as Unit, Null, Nothing, Any, AnyRef
    • To achieve the "everything is an object" idea, Scala wraps the primitives in a corresponding Rich* class; for example, RichDouble
    • Scala also has support for Tuple, List, Range, and Map literals
  • Python
    • Python has Number, Boolean, String, Regex, Tuple, List, and Dict literals
    • Python also supports Objects and Classes

Syntax considerations

  • OSM
    • Being Funny...Otherwise the syntax is incredibly verbose
  • Scala
    • Devised to be incredibly concise.
    • For example, complex idea of mutablility brought down to one word: val or var
  • Python
    • Python, much like Scala, is meant to be concise
    • Significant whitespace and short builtin function names make the syntax both easy to read and write (i.e lines of code are not very dense syntactically)

Support for Abstraction

  • OSM
    • Through use of classes, we can abstract away the complexities of what is being done to the underlying data structures
    • The idea of "all things are functions" allows for a very expressize DSL allowing for an even further abstraction of what is really happening
  • Scala
    • We stole our idea from Scala, so see above point again :P
  • Python
    • Python allows the common level of OOP abstraction through the use of classes and modules of functions

Expressivity

  • OSM
    • This really relates to our previous point: By allowing for a high level of abstraction, we can allow the developer to be as expressive as they want
  • Scala
    • Again, we stole this idea from Scala
    • However, Scala's attempt to blend functional and object-oriented styles allows us to be even more expressive in what we are doing
  • Python
    • Complex functional operations and various literals allow for complex operations to be expressed in a small amount of code
    • For example, the list comprehensions we showed earlier

Type Checking

  • OSM
    • Heavy use of static typing, as seen in how our syntax requires all variables to be given a type explicitly
  • Scala
    • Scala uses static typing, but its syntax is far more forgiving than ours
    • val temp = "String" and val temp : String = "String" mean the same thing
    • It will attempt to infer the given type in the event it is not given
  • Python
    • Dynamic as dynamic can be
    • How Dynamic is it?
      • age = 5
age = "Some String"
age = [ 0, 1, 2, 3]

Exception Handling

  • OSM
    • Don't have it!
    • So, the other two are better, no need to compare :)
